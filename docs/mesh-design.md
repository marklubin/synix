# synix mesh: Distributed Build & Deploy

## Context

Synix is a build system for agent memory. Currently, builds run on one machine. **synix mesh** adds a distribution layer: machines form a named mesh, a central server receives source data, runs builds, and distributes artifacts back. Like Tailscale but for synix pipelines.

**Everything lives in the synix monorepo** (`/home/mark/synix/`):
- `src/synix/mesh/` — the distribution library
- `src/synix/cli/mesh_commands.py` — CLI: `synix mesh create`, `synix mesh server`, etc.
- `src/synix/templates/06-claude-sessions/` — claude-sync as a synix template
- `tests/e2e/mesh/` — Podman Compose E2E test harness

## Design Constraints & Assumptions

### Transport: Tailscale-only
Mesh traffic is assumed to run over a Tailscale network (or equivalent private overlay). The server binds to `0.0.0.0` but is only reachable via Tailscale IPs. Internet-routable deployments are **not supported** in v1.

### Authentication: Shared mesh token + node identity
Despite the private network assumption, all API requests require a shared secret token (`mesh_token` in config, passed via `Authorization: Bearer <token>` header). This prevents accidental cross-mesh interference and provides a defense-in-depth layer. The token is generated during `mesh create` and copied to all members during `provision`.

```toml
[mesh]
name = "marks-claude"
token = "msh_a1b2c3..."   # generated by mesh create, 32-byte random hex
```

All server endpoints (except `/api/v1/health`) reject requests without a valid token (HTTP 401).

All requests also include an `X-Mesh-Node: <hostname>` header. The server records this in session metadata and heartbeat state for audit attribution. This is not a security boundary (a node can lie about its identity with the shared token), but provides operational traceability for debugging.

**Deferred to v2**: Per-node credentials with individual revocation.

**v1 risk acceptance & token rotation runbook**: A leaked token compromises the full mesh until rotated. To rotate:
```bash
# On any node — generates new token, writes to local config
synix mesh rotate-token --name "marks-claude"
# Then re-provision each node (copies new config)
synix mesh provision --name "marks-claude" --role server   # on server
synix mesh provision --name "marks-claude" --role client --server obispo:7433  # on each client
```
Rotation requires brief downtime (clients reject until re-provisioned). The old token is immediately invalid on the server after its re-provision.

### Durability: Best-effort with client re-submit
Server state (`sessions.db` + compressed session files) is **leader-local and not replicated**. If the leader dies and storage is lost, ingested sessions that no client still has locally are gone. This is acceptable because:
- Source files originate on client machines and persist in `watch_dir` until explicitly deleted
- On leader failover, clients re-scan `watch_dir` and re-submit all sessions to bootstrap the new leader
- Artifacts (build outputs) are pulled by clients and survive leader loss

**Durability by phase:**
- **Unsent local files** (exist in `watch_dir`, not yet submitted): safe — persist on client disk, will be submitted on next scan cycle. Bounded by `scan_interval`.
- **Sent-but-not-built** (ingested by server, in `sessions.db` + compressed store): lost on leader disk failure. Recoverable if the originating client still has the file in `watch_dir` (clients re-submit on failover).
- **Built artifacts** (in bundle, pulled by clients): survive leader loss — clients retain their last-pulled bundle locally.

**RPO**: zero for files still in any client's `watch_dir` (the common case). Non-zero only for files that were submitted, then deleted from the originating client before failover — those are unrecoverable. **RTO**: `leader_timeout` + election + full rebuild on new leader.

### Deploy hooks: trusted-local only
Deploy hook commands (`[deploy.server]` and `[deploy.client]`) execute as the user running the mesh daemon. No sandboxing. The config file is the trust boundary — if you can edit `synix-mesh.toml`, you can run arbitrary commands.

## Onboarding Story

```bash
# 1. Create a mesh (on any machine — generates config)
uvx synix mesh create --name "marks-claude" --pipeline ./pipeline.py
#   → creates synix-mesh.toml with defaults
#   → prompts for source watch_dir, patterns, deploy hooks

# 2. Provision this machine as the server
uvx synix mesh provision --name "marks-claude" --role server
#   → creates data dirs (~/.synix-mesh/marks-claude/)
#   → installs + starts systemd service (synix-mesh-server@marks-claude)
#   → shows: "Server running on 0.0.0.0:7433"

# 3. On another machine: provision as client
uvx synix mesh provision --name "marks-claude" --role client --server obispo:7433
#   → creates client state dir
#   → installs + starts systemd service (synix-mesh-client@marks-claude)
#   → shows: "Client connected to obispo:7433, submitting 42 existing sessions..."

# 4. Check mesh status
uvx synix mesh status --name "marks-claude"
#   → shows: leader, members, last build, artifact count, health

# 5. Manual operations
uvx synix mesh submit --name "marks-claude" ./session.jsonl    # one-shot submit
uvx synix mesh build --name "marks-claude"                     # force a build
uvx synix mesh deploy --name "marks-claude"                    # force local deploy
```

|--------|----------|------|---------|

| GET | `/api/v1/artifacts/manifest` | Yes | Manifest hash (ETag) |
| GET | `/api/v1/artifacts/bundle` | Yes | Download artifact bundle (304 if unchanged) |
| POST | `/api/v1/search` | Yes | Query search index |

| GET | `/api/v1/status` | Yes | Detailed server stats: build count, session count, uptime, version |

## Server Internals

### Session Store (`mesh/store.py`)
- SQLite `sessions.db`: `(session_id, project_dir, submitted_by, jsonl_sha256, submitted_at, processed)`
- Compressed files: `sessions/{project_dir}/{session_id}.jsonl.gz`
- Dedup via sha256

### BuildScheduler (`mesh/scheduler.py`)

State machine with single-flight enforcement:

```
  ┌─────────┐   pending sessions    ┌────────┐   threshold/timeout   ┌─────────┐
  │  IDLE   │ ──────────────────► │ QUEUED │ ──────────────────► │ RUNNING │
  └─────────┘                      └────────┘                      └────┬────┘
       ▲                                ▲                               │
       │         build complete         │    new sessions during build  │
       └────────────────────────────────┴───────────────────────────────┘
                                   (coalesce into next QUEUED)
```

- **IDLE**: no pending sessions, no active build
- **QUEUED**: pending sessions accumulating; transitions to RUNNING when `pending >= batch_threshold` OR `time_since_first_pending >= max_delay`
- **RUNNING**: build in progress via `asyncio.to_thread()`; new sessions arriving during build go to QUEUED (coalesced)
- Only one build runs at a time
- `builds/trigger` during RUNNING: sets a `force_rebuild` flag, returns **202 Accepted** with `{"status": "queued", "message": "Build in progress, rebuild queued on completion"}`. The CLI retries with exponential backoff (1s, 2s, 4s) until the queued build completes or 120s timeout.
- After build completes: `[deploy.server]` hooks → `package()` → update search → notify
- If QUEUED or `force_rebuild` flag set during completion, immediately starts next build

### GenericServerSource (`mesh/source.py`)
- `Source` subclass reading from server's compressed store
- Server sets `env_var` (from config) before build → pipeline Source reads from it

## Client Internals

Three async loops:
1. **Watcher** (scan_interval): scan watch_dir → sha256 → submit new files
2. **Puller** (pull_interval): ETag check → pull bundle → run `[deploy.client]` hooks
3. **Heartbeat** (heartbeat_interval): POST → 3 failures = leader election

---

## Leader Election

Term-based priority election with `(term, leader_id)` fencing:

### Election config invariant
All nodes must have identical `[cluster].leader_candidates` ordering. A `cluster_config_hash` (SHA-256 of the ordered candidate list, serialized as `join("\0", candidates)`) is computed at startup and included in every heartbeat and `/api/v1/cluster` response. The hash preserves insertion order — two nodes with the same candidates in different priority order will produce different hashes. The server rejects heartbeats with a mismatched hash (HTTP 409 with `{"error": "cluster_config_mismatch", "expected": "...", "got": "..."}`). `synix mesh status` displays the hash for operator visibility.

### Election protocol

1. **Term = `(counter, leader_id)`**: monotonically increasing counter plus the hostname of the leader that claimed it. Persisted in `state.json`. All API responses include both fields. All client requests include both fields.
2. **Detection**: client misses 3 consecutive heartbeats (`leader_timeout`) → enters election mode
3. **Candidate selection**: iterate `[cluster].leader_candidates` list in order, `tailscale ping` each, first alive wins
4. **Promotion**: winner increments counter in its `state.json`, stamps its own hostname as `leader_id`, starts server: `systemctl --user start synix-mesh-server@{name}`
5. **Fencing**: server rejects any write request where:
   - request term counter < server's term counter, OR
   - request term counter == server's term counter AND request `leader_id` != server's hostname
6. **Equal-term tie-breaking**: if two candidates both increment from term `N` to `N+1` (race during partition), clients resolve the conflict deterministically: the candidate with the **lower index** in `leader_candidates` is the valid leader for that term. Clients ignore the other. The losing leader will receive heartbeats stamped with the winning `leader_id` and step down.
7. **Old leader step-down**: if a running server receives a request with a higher term counter, OR same counter but a higher-priority `leader_id`, it stops accepting writes, shuts down, and rejoins as client.
8. **Leader self-check**: the server runs a periodic self-check (every `heartbeat_interval`). It pings higher-priority candidates in the `leader_candidates` list. If a higher-priority node is alive and claims the same or higher term, the server fails closed: stops accepting writes and initiates step-down. This ensures convergence even when client traffic is sparse.
9. **Bootstrap**: clients re-scan `watch_dir` and re-submit all sessions to the new leader

```
Term (1, obispo):  obispo (leader)     salinas (client)     oxnard (client)
                    │                       │                     │
                    ✕ dies                  │                     │
                                       3 missed HB            3 missed HB
                                       election →             election →
                                       ping obispo: fail      ping obispo: fail
                                       ping salinas: ok!      (salinas already won)
                                            │
Term (2, salinas):  salinas (leader)    clients re-submit     oxnard (client)
```

**Split-brain prevention**: In a partition where both salinas and oxnard promote to term `(2, salinas)` and `(2, oxnard)` respectively, clients deterministically pick salinas (lower candidate index). Oxnard's server receives heartbeats stamped `(2, salinas)` — salinas has higher priority at equal term — and steps down. Sessions submitted to oxnard during the brief split are re-submitted by clients to salinas during bootstrap.
