# Synix Mesh (Experimental)

> **Warning:** Synix Mesh is experimental. Commands, configuration format, and behavior may change in future releases.

Synix Mesh distributes pipeline builds across machines. A central server receives source files from clients, runs builds, and distributes artifact bundles back. Clients watch local directories for new source files and automatically submit them. The server schedules builds, packages results, and serves them to clients on demand.

Mesh assumes all machines are connected via [Tailscale](https://tailscale.com/) (or an equivalent private network). Internet-routable deployments are not supported.

## Quick Start

```bash
# Install mesh dependencies (not included in base synix)
pip install synix[mesh]

# 1. Create a mesh — generates config and shared token
uvx synix mesh create --name my-mesh --pipeline ./pipeline.py

# 2. Provision this machine as the server
uvx synix mesh provision --name my-mesh --role server

# 3. On another machine: provision as a client
uvx synix mesh provision --name my-mesh --role client --server server-hostname:7433

# 4. Check status
uvx synix mesh status --name my-mesh
```

After provisioning, both server and client run as systemd user services. The client watches its configured directory for new source files, submits them to the server, and pulls artifact bundles after each build.

## How It Works

```
  Client (any machine)                 Server (leader)
  ════════════════════                 ═══════════════
  source files                         HTTP API on 0.0.0.0:7433
      │                                    │
  Watcher (30s)                         ┌──▼──────────────┐
      │ POST /sessions                  │  Session Store   │
      ├──────────────────────────────►  │  sessions.db     │
      │                                 │  sessions/*.gz   │
      │                                 └──────┬──────────┘
      │                                        │
      │                                 ┌──────▼──────────┐
      │                                 │  BuildScheduler  │
      │                                 │  (debounced)     │
      │                                 │  synix build     │
      │                                 │  package + deploy│
      │                                 └──────┬──────────┘
      │                                        │
  Puller (60s)                          ┌──────▼──────────┐
      │ GET /artifacts/bundle           │  Artifact Server │
      ◄────────────────────────────────┤  Search API      │
      │                                 │  Cluster mgmt   │
      ▼                                 └─────────────────┘
  deploy(client hooks)

  Heartbeat (120s) ──────────────────►  Cluster state
      └── 3 missed = elect new leader
```

**Clients** run three async loops:
1. **Watcher** — scans a local directory for new files, computes SHA-256 hashes, and submits new files to the server.
2. **Puller** — periodically checks for new artifact bundles via ETag. Downloads and extracts when changed, then runs deploy hooks.
3. **Heartbeat** — posts to the server at regular intervals. Three consecutive failures trigger leader election.

**The server** receives source files, stores them in a compressed SQLite-backed store with SHA-256 deduplication, and schedules builds. Builds are debounced: the scheduler waits for a batch of files or a timeout before triggering. After a build completes, the server packages artifacts into a tar.gz bundle, runs server-side deploy hooks, and serves the bundle to clients.

## Configuration

Mesh configuration lives in `synix-mesh.toml`, generated by `mesh create` and stored in `~/.synix-mesh/{name}/`.

```toml
[mesh]
name = "my-mesh"
token = "msh_a1b2c3d4e5f6..."            # generated automatically

[pipeline]
path = "./pipeline.py"                    # pipeline file (relative or absolute)

[source]
watch_dir = "~/.claude/projects"          # directory clients watch for new files
patterns = ["**/*.jsonl"]                 # glob patterns to match
exclude = ["**/subagents/**"]             # patterns to skip

[server]
port = 7433                               # HTTP port
build_min_interval = 300                  # minimum seconds between builds
build_batch_threshold = 5                 # trigger build immediately at N pending files
build_max_delay = 900                     # maximum seconds to wait before building

[client]
scan_interval = 30                        # seconds between directory scans
pull_interval = 60                        # seconds between bundle checks
heartbeat_interval = 120                  # seconds between heartbeats

[cluster]
leader_candidates = ["obispo", "salinas"] # priority-ordered failover list
leader_timeout = 360                      # seconds before declaring leader dead

[bundle]
include = ["manifest.json", "context.md", "layer2-*/**", "layer3-*/**"]
exclude = ["search.db", "*.lock"]

[deploy.server]
commands = ["python post_build.py --build-dir {build_dir}"]

[deploy.client]
commands = ["python deploy.py --build-dir {build_dir}"]

[notifications]
webhook_url = "http://localhost:8080/notifications/push"
```

### Environment Variable Overrides

| Variable | Overrides |
|----------|-----------|
| `SYNIX_MESH_ROOT` | Root directory for all meshes (default: `~/.synix-mesh`) |
| `SYNIX_MESH_PORT` | Server port |
| `SYNIX_MESH_SOURCE_WATCH_DIR` | Client watch directory |

## CLI Reference

All mesh commands are under the `synix mesh` subgroup.

| Command | Purpose |
|---------|---------|
| `uvx synix mesh create --name NAME --pipeline PATH` | Create a new mesh (generates config and token) |
| `uvx synix mesh provision --name NAME --role server\|client [--server HOST:PORT]` | Provision this machine as a member |
| `uvx synix mesh server --name NAME` | Start the server daemon |
| `uvx synix mesh client --name NAME` | Start the client daemon |
| `uvx synix mesh status --name NAME` | Show mesh health, members, and last build info |
| `uvx synix mesh submit --name NAME FILE` | One-shot source file submission |
| `uvx synix mesh build --name NAME` | Force trigger a build on the server |
| `uvx synix mesh deploy --name NAME` | Force run local deploy hooks |
| `uvx synix mesh list` | List all meshes on this machine |
| `uvx synix mesh rotate-token --name NAME` | Generate a new mesh token |
| `uvx synix mesh destroy --name NAME` | Remove mesh config, data, and stop services |

### Provisioning

Provisioning creates the data directory structure, copies the config, and installs a systemd user service.

```bash
# Server — starts listening for submissions and running builds
uvx synix mesh provision --name my-mesh --role server

# Client — needs the server address
uvx synix mesh provision --name my-mesh --role client --server obispo:7433
```

After provisioning, the daemon starts automatically and persists across reboots via `systemctl --user enable`.

### Manual Operations

You don't need to wait for the client watcher cycle. Submit files directly:

```bash
uvx synix mesh submit --name my-mesh ./session.jsonl
```

Force a build on the server (useful after manual submissions or to reprocess with an updated pipeline):

```bash
uvx synix mesh build --name my-mesh
```

If a build is already in progress, the command returns immediately with a "queued" status. The queued build runs automatically after the current one finishes.

## Server API

The server exposes an HTTP API on the configured port. All endpoints except `/api/v1/health` require `Authorization: Bearer <token>`.

| Method | Endpoint | Purpose |
|--------|----------|---------|
| POST | `/api/v1/sessions` | Submit a source file (base64-encoded, SHA-256 dedup) |
| GET | `/api/v1/builds/status` | Current build state (idle/queued/running) |
| POST | `/api/v1/builds/trigger` | Manually trigger a build (202 if already running) |
| GET | `/api/v1/artifacts/bundle` | Download the artifact bundle (304 if unchanged) |
| POST | `/api/v1/search` | Query the search index |
| POST | `/api/v1/heartbeat` | Client heartbeat with term and config hash |
| GET | `/api/v1/cluster` | Cluster state: leader, members, terms |
| GET | `/api/v1/health` | Liveness check (unauthenticated) |
| GET | `/api/v1/status` | Detailed stats: build count, sessions, uptime |

## Build Scheduling

The server doesn't build on every submission. A debounce scheduler manages build timing:

- **Minimum interval** (`build_min_interval`): minimum seconds between builds
- **Batch threshold** (`build_batch_threshold`): if this many files are pending, build immediately
- **Maximum delay** (`build_max_delay`): never wait longer than this to start a build

Only one build runs at a time. If new files arrive during a build, they queue for the next run. Triggering a build via `mesh build` during an active build returns HTTP 202 and queues a follow-up.

## Deploy Hooks

After a build completes (server) or a bundle is pulled (client), deploy hooks run. These are shell commands defined in `[deploy.server]` and `[deploy.client]`.

The `{build_dir}` placeholder is substituted with the actual build output directory:

```toml
[deploy.server]
commands = ["python post_build.py --build-dir {build_dir}"]

[deploy.client]
commands = ["cp {build_dir}/context.md ~/.claude/context.md"]
```

Hooks run as the user running the daemon. If a hook exits non-zero, the failure is logged and the build is not published as a bundle. This is by design — a failed post-processing step should not distribute broken artifacts.

## Leader Election and Failover

Mesh supports automatic failover when the server becomes unreachable.

### How It Works

1. Clients send heartbeats to the server. After 3 consecutive failures, they enter election mode.
2. Candidates are tried in priority order from `[cluster].leader_candidates`. The first reachable candidate wins.
3. The winner increments the term counter, promotes itself to server, and starts accepting submissions.
4. Clients detect the new leader and re-submit all files from their watch directory.

### Term-Based Fencing

Every election increments a term counter. The server rejects requests from clients with a stale term, preventing split-brain scenarios where two nodes briefly both think they're the leader.

If two candidates promote simultaneously (network partition), clients deterministically pick the one with the lower index in `leader_candidates`. The other detects this via incoming heartbeats and steps down.

### Config Hash Validation

All nodes must have identical `leader_candidates` ordering. A SHA-256 hash of the candidate list is included in every heartbeat. If a client's hash doesn't match the server's, the heartbeat is rejected with HTTP 409. This catches configuration drift before it causes election confusion.

### Leader Self-Check

The server periodically probes higher-priority candidates. If a higher-priority node comes back online and claims the same or higher term, the current leader steps down. This ensures the highest-priority available node eventually becomes leader.

## Security Model

Mesh relies on two layers of security:

1. **Network isolation** — All traffic runs over Tailscale (or equivalent). The server binds to `0.0.0.0` but is only reachable via private network IPs.
2. **Shared token** — Every API request (except health checks) requires a Bearer token generated during `mesh create`. This prevents accidental cross-mesh interference.

Requests also include an `X-Mesh-Node` header with the client's hostname for audit attribution.

### Token Rotation

If a token is compromised, rotate it:

```bash
# Generate new token and update local config
uvx synix mesh rotate-token --name my-mesh

# Re-provision every node (copies new config)
uvx synix mesh provision --name my-mesh --role server        # on server
uvx synix mesh provision --name my-mesh --role client --server HOST:PORT  # on each client
```

Rotation requires brief downtime — clients are rejected until re-provisioned with the new token.

### Limitations

- Tokens are shared secrets, not per-node credentials. A leaked token compromises the entire mesh until rotated.
- Deploy hooks execute as the daemon user with no sandboxing. The config file is the trust boundary.
- Per-node credentials with individual revocation are planned for a future release.

## Durability

Server state (session database + compressed files) is **not replicated**. If the leader's disk fails, ingested sessions that no client still has locally are lost. In practice this rarely matters:

- **Unsent files** (in client's `watch_dir`): safe — persist on client disk.
- **Sent-but-not-built** (in server's session store): recoverable if the originating client still has the file. Clients re-submit everything on failover.
- **Built artifacts** (pulled by clients): survive leader loss — clients keep their last-pulled bundle.

The durability model is: clients are the source of truth. As long as source files remain in a client's watch directory, they can always be resubmitted.

## Data Layout

```
~/.synix-mesh/
└── my-mesh/                       # one directory per mesh
    ├── synix-mesh.toml            # configuration
    ├── pipeline.py                # pipeline definition
    ├── state.json                 # role, server URL, hostname, term, leader
    ├── server/                    # server role only
    │   ├── sessions/              # submitted source files (gzipped)
    │   ├── sessions.db            # session metadata (SQLite)
    │   ├── build/                 # synix build output
    │   └── bundles/               # pre-built tar.gz artifacts
    └── client/                    # client role only
        ├── submitted_sessions.json  # SHA-256 hashes of submitted files
        └── build/                 # pulled and extracted artifacts
```

## Notifications

The server can send webhook notifications on build events (completion, failure). Configure in `[notifications]`:

```toml
[notifications]
webhook_url = "http://localhost:8080/notifications/push"
source = "my-mesh"
```

Notifications are best-effort — failures are logged but never block builds.

## Installation

Mesh dependencies (starlette, uvicorn, httpx) are optional extras, not included in base synix:

```bash
pip install synix[mesh]
# or with uv
uv pip install synix[mesh]
```

If you install synix without the `[mesh]` extra, the `synix mesh` commands will import-error with a clear message about missing dependencies.
